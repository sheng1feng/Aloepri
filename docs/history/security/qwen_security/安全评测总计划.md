# Qwen 安全评测总计划

## 1. 目标

本计划聚焦回答四个问题：

1. 当前仓库的 `Qwen` 路线在**不考虑开发态 hook / recorder 暴露**时，面对论文中的典型攻击还剩多少泄露；
2. 当前仓库的 `Qwen` 路线在**客户端 / 服务端分离部署**时，若服务端可见有限中间态，攻击面会扩大到什么程度；
3. `Stage H` 研究线与 `Stage J / Stage K` 标准形状交付线，安全性是否存在明显差距；
4. 目前最主要的薄弱面是**权重侧 permutation 恢复**、**训练式 embedding 反演**、**中间态反演**，还是**token 频率利用**；
5. 在尽量不破坏现有部署形态的前提下，哪些增强点最值得优先验证。

## 2. 当前范围

### In scope

- 基于 `Qwen` 的安全评测基线建立
- 论文攻击中与当前约束匹配的子集：
  - `VMA`
  - `IA`
  - `NN / IMA`
  - `ISA`
  - `TFMA`
  - `SDA`
- `stable_reference` / `tiny_a` / 必要扩展 profile 的对比
- `Stage H`、`Stage J`、`Stage K` 三条代表性工件的对比分析
- 参数与安全性的趋势扫描
- 部署场景下服务端可能观察到的中间态攻击面分析：
  - attention score
  - decoder hidden states
  - KV cache / layer output
  - logits 前局部表示

### Out of scope（当前轮）

- 仅由本机开发时的 `hook / recorder / debug trace` 暴露导致的攻击
- 安全多方计算 / FHE / TEE 类替代路线
- Llama 安全评测

### 威胁模型补充

当前轮次把“中间态攻击”限定为**部署态中间态攻击**：

- client 与 server 分离；
- server 不持有明文 prompt / 明文输出；
- 但 server 可能在真实部署中接触到部分推理中间信息，例如：
  - attention score
  - 某些层 hidden state
  - KV cache
  - 推理服务日志中暴露的局部张量摘要

当前轮次不把以下情形计入攻击能力：

- 本机调试时人为插入的 hook
- 本地 recorder / trace 文件
- 仅开发环境才存在的 debug dump

## 3. 评测对象

首轮统一使用以下对象：

### 3.1 工件对象

- `artifacts/stage_h_full_obfuscated`
- `artifacts/stage_j_full_square`
- `artifacts/stage_j_full_square_tiny_a`
- `artifacts/stage_k_release`

### 3.2 profile 对象

- `stable_reference`
- `tiny_a`

如需要补实验点，再增加：

- `paper_default` 风格参数
- 更弱噪声点
- 更强噪声点

### 3.3 数据对象

- 先用仓库已有 prompt 与小规模本地文本集做 smoke
- 再引入一份公开文本集，专门服务 `NN / IMA / TFMA / SDA`
- 动态攻击与频率攻击的数据集需要固定 train / attack / eval 切分

## 4. 评测指标

### 4.1 功能侧指标

- `generated_ids_exact_match_rate`
- `generated_text_exact_match_rate`
- `greedy_first_token_match_rate`
- `avg_full_logits_max_abs_error`
- `avg_full_logits_mean_abs_error`

### 4.2 安全侧指标

- `TTRSR`：Token Top-k / Top-1 恢复成功率
- `PIIRSR`：对敏感 token 的恢复成功率
- `BLEU-4`：恢复文本质量
- `CosSim`：向量恢复相似度
- attack-vs-baseline 的相对提升/下降
- 中间态可分离性 / 可逆性指标
- 从中间态恢复 token 的 Top-k 成功率

### 4.3 工程侧指标

- 攻击运行时间
- 攻击对数据规模的敏感性
- 攻击对 profile / 参数点的敏感性

## 5. 分阶段计划

## Phase 0：安全基线与实验约定

目标：

- 明确威胁模型、攻击输入、评测输出 schema
- 固定工件、profile、数据集、指标
- 把“研究线”和“部署线”拆开评估
- 把“静态权重攻击 / 部署态中间态攻击 / 在线频率攻击”拆成三张独立矩阵

交付：

- 统一攻击结果 JSON schema
- 攻击实验目录规范
- 首版评测矩阵

完成标志：

- 后续所有攻击脚本都能输出同一套 summary 字段

---

## Phase 1：VMA（权重侧 permutation 恢复）

优先级：`P0`

原因：

- 最贴近当前仓库的核心秘密：`token permutation`
- 最能直接检验 embedding/head 噪声是否足够
- 对 `Stage J / K` 特别关键

目标：

- 在 `Stage H / J / K` 工件上建立 VMA 基线
- 比较 `stable_reference` 与 `tiny_a` 的恢复率
- 验证“噪声不足时恢复率上升”的趋势

重点输出：

- token 恢复率
- 敏感 token 恢复率
- 各权重组合的贡献度
- voting 前后差异

完成标志：

- 能回答“当前最危险的权重组合是什么”

---

## Phase 2：IA（结构不变量攻击）

优先级：`P1`

原因：

- 这是对“不是纯置换”的进一步检验
- 可以帮助判断现有实现中是否仍残留可利用不变量

目标：

- 先做 `Gate-IA`
- 再做与 attention 相关但**不依赖中间态**的 `Attn-IA`
- 判断 `Stage H` 与 `Stage J / K` 哪条线更容易留下不变量

重点输出：

- 不变量统计量的可分离性
- token 恢复率
- 噪声强度与攻击成功率关系

完成标志：

- 能回答“当前实现最容易保留的 invariants 在哪一层”

---

## Phase 3：NN / IMA（训练式 embedding 反演）

优先级：`P0`

原因：

- 更贴近“攻击者有训练资源”的现实场景
- 能直接反映 obfuscated embedding 是否仍保留较强明文语义

目标：

- 训练轻量反演器，从 obfuscated embedding 预测：
  - 原 token id
  - 原 embedding
- 比较 `Stage A`、`Stage H`、`Stage J`
- 验证 `tiny_a` 是否显著优于 `stable_reference`

重点输出：

- Top-1 / Top-k token 恢复率
- embedding cosine similarity
- 不同训练集规模下的恢复曲线

完成标志：

- 能回答“当前最危险的不是静态匹配，而是训练式反演吗”

---

## Phase 4：ISA（部署态中间态反演）

优先级：`P0`

原因：

- 这是论文里最关键的“服务端看到内部计算结果后是否能反推输入”问题；
- 你现在明确要求考虑 client/server 分离后的真实部署中间态攻击；
- 它最能区分“研究线”和“部署线”之间的安全差距。

目标：

- 不依赖本机 hook，而是基于“部署态可见中间量”定义攻击接口；
- 先做两类对象：
  - attention score
  - decoder hidden states / layer outputs
- 比较：
  - 仅 noise
  - noise + transform
  - 加/不加额外 permutation

重点输出：

- 从中间态恢复 token 的 Top-1 / Top-k 成功率
- 不同层、不同中间量的攻击难度
- `Stage H / J / K` 在中间态攻击下的差异

完成标志：

- 能回答“当前部署线最危险的中间态到底是什么”

---

## Phase 5：TFMA / SDA（频率利用攻击）

优先级：`P1`

原因：

- 这是 deterministic token permutation 的天然弱点
- 更贴近长期在线观测场景

目标：

- 先做 TFMA
- 再做 SDA
- 评估零知识 / 域知识 / 分布知识三种设置

重点输出：

- Top-10 / Top-100 token 恢复率
- BLEU-4
- 数据量增长后的攻击趋势

完成标志：

- 能回答“频率侧泄露在当前 Qwen release 工件下有多严重”

---

## Phase 6：参数-安全性联合扫描

优先级：`P0`

原因：

- 当前仓库已做过大量 correctness 定标，但还缺 privacy curve

目标：

- 联合扫描：
  - `alpha_e`
  - `alpha_h`
  - 必要时的 `lambda`
- 画出 accuracy / security tradeoff

重点输出：

- 不同参数点下的：
  - correctness
  - VMA 恢复率
  - IMA 恢复率
  - TFMA/SDA 恢复率

完成标志：

- 能形成首版 `Qwen` 推荐安全工作点

---

## Phase 7：增强点验证

优先级：`P1`

目标不是重新发明 AloePri，而是在当前仓库基础上寻找高性价比增强点。

建议优先验证：

1. 更细粒度的 `alpha_e / alpha_h` 工作点
2. 针对频率攻击的轻量平滑策略
3. profile 分层交付策略
4. secret 生命周期管理策略
5. 研究线与部署线分离后的安全声明边界

增强点以“不明显破坏当前 HF / Stage K 交付形态”为优先约束。

## 6. 建议的安全工作流

### 工作流 A：先复现论文攻击

顺序：

1. `VMA`
2. `NN / IMA`
3. `ISA`
4. `IA`
5. `TFMA / SDA`

### 工作流 B：再做当前仓库差距分析

对比：

- 论文原始结论
- `Stage H` 研究线
- `Stage J / K` 部署线

### 工作流 C：最后做增强点实验

输出：

- 推荐 profile
- 风险说明
- 加强建议

## 7. 当前建议的里程碑

### Milestone 1：安全评测框架跑通

- 攻击结果 schema 定稿
- `VMA` 首版脚本可跑
- `NN / IMA` 首版训练脚本可跑

### Milestone 2：首版 Qwen 安全报告

- 有 `Stage H / J / K` 三线对比
- 有 `stable_reference / tiny_a` 对比
- 有至少三类攻击结果
- 有一类部署态中间态攻击结果

### Milestone 3：首版增强建议

- 形成“当前最危险攻击面”
- 形成“当前推荐工作点”
- 形成“下一个最值得做的增强项”

## 8. 一句话结论

当前 `Qwen` 安全工作应优先回答：

> **在考虑 client/server 分离部署后可见中间态的前提下，当前仓库的标准交付线是否比研究线更容易遭受 permutation 恢复、embedding 反演、中间态反演与频率利用攻击；以及应该如何在尽量不破坏部署形态的前提下提升安全性。**
