### 当前架构支持边界

仓库当前的模块化层不再把模型写死为 `Qwen2.5-0.5B-Instruct`，而是通过 **Qwen decoder 结构契约** 做适配。

也就是说，只要模型满足以下结构：

- `model.embed_tokens`
- `model.layers`
- `model.norm`
- `lm_head`
- 每层具备：
  - `self_attn.q_proj / k_proj / v_proj / o_proj`
  - `input_layernorm / post_attention_layernorm`
  - `mlp.gate_proj / up_proj / down_proj`

就可以进入当前 `src/aloepri/` 的模块化主线。

当前已实际验证的 checkpoint 仍然是：

- `Qwen2.5-0.5B-Instruct`

但模块接口本身已经不依赖固定 hidden size、层数或 head 数。

## 核心概念与数据流

### 目标：协变混淆（Covariant Obfuscation）

技术报告把推理抽象为：

- 明文推理：`f(x, θ) -> y`
- 协变混淆：在混淆空间里推理 `f~(ϕX(x), ϕΘ(θ)) -> ỹ`，并通过 `ψY` 还原输出

在工程上，这拆解为两条链路：

- 离线：构造混淆后的模型权重（embedding/head + attention + ffn + norm），并导出服务端与客户端需要的产物
- 在线：客户端映射 token（输入/输出），服务端按“尽量不改推理框架”的方式推理

### 本仓库的实现路线

仓库主线把“最终整体 AloePri”拆成可验证的子闭环：

- Stage A：词表置换闭环（token 映射 + embedding/head 置换）
- Stage B/C/D/E：从单层 attention/ffn/norm 到多层传播与复杂 attention（RoPE block、tau_kv/tau_group）
- Stage F：引入论文 Algorithm 1 的 KeyMat 体系（可逆扩维变换）
- Stage G：将 wrapper 逻辑融合进权重（推理时更贴近原模型执行路径）
- Stage H：Attention 静态化与噪声定标（更接近“可部署”的形态）

这些阶段的“实验报告”在 `docs/` 中；本技术文档更关注模块如何组织、如何调用、如何扩展。

## 模块详解（src/）

### 1) 工具与评估

- 模型加载、prompt 格式化、tokenize： [model_loader.py](file:///home/shengfeng/Privacy-inference/src/model_loader.py)
- 误差指标与输出采集： [evaluator.py](file:///home/shengfeng/Privacy-inference/src/evaluator.py)
- Token/Logits 映射工具： [transforms.py](file:///home/shengfeng/Privacy-inference/src/transforms.py)

常见用法：

- `map_input_ids(...)`：输入 token 映射到混淆 token
- `restore_logits(...)`：将混淆空间 logits 还原回明文词表顺序（注意：该函数的语义是“按 perm_vocab 重新索引”，与 `inv_perm_vocab` 的用法不同）

### 2) Token 置换与 Stage A

- 生成“可移动 token id”集合与词表置换： [key_manager.py](file:///home/shengfeng/Privacy-inference/src/key_manager.py)
- Stage A 入口与模型拷贝/置换： [stage_b.py:prepare_stage_a_model](file:///home/shengfeng/Privacy-inference/src/stage_b.py#L225-L237)
- 直接对 embedding/head 权重做行重排： [obfuscate_embed_head.py](file:///home/shengfeng/Privacy-inference/src/obfuscate_embed_head.py)

设计要点：

- 置换不是全量 token indiscriminate shuffle，需对 special tokens 等保持稳定（具体策略在 Stage A 相关实现中）

### 3) Hidden-space 变换（Stage B/C/D 的基础）

- Hidden permutation + scaling 的最小协变变换： [hidden_keys.py](file:///home/shengfeng/Privacy-inference/src/hidden_keys.py)
- RMSNorm 的 kappa 修正（HiddenTransform 版本）： [obfuscate_rmsnorm.py](file:///home/shengfeng/Privacy-inference/src/obfuscate_rmsnorm.py)
- FFN 的置换+缩放（HiddenTransform 版本）： [obfuscate_ffn.py](file:///home/shengfeng/Privacy-inference/src/obfuscate_ffn.py)

### 4) 复杂 Attention（Stage E/H 的关键）

- RoPE block、R_qk、H_qk、Z_block、tau_kv/tau_group 的生成： [attention_keys.py](file:///home/shengfeng/Privacy-inference/src/attention_keys.py)
- GQA 头布局与分组操作： [gqa_layout.py](file:///home/shengfeng/Privacy-inference/src/gqa_layout.py)
- 复杂 attention wrapper（用于对齐/排错/trace）： [obfuscate_attention_complex.py](file:///home/shengfeng/Privacy-inference/src/obfuscate_attention_complex.py)
- Attention 静态化（权重融合、减少 wrapper 依赖）： [stage_h_attention_static.py](file:///home/shengfeng/Privacy-inference/src/stage_h_attention_static.py)

### 5) KeyMat（论文 Algorithm 1）与后续融合

- Algorithm 1 的 KeyMat 生成与逆矩阵校验： [keymat.py](file:///home/shengfeng/Privacy-inference/src/keymat.py)
- KeyMat 的 embedding/head 侧融合： [keymat_embed_head.py](file:///home/shengfeng/Privacy-inference/src/keymat_embed_head.py)
- KeyMat 的 RMSNorm/FFN 融合工具： [keymat_norm.py](file:///home/shengfeng/Privacy-inference/src/keymat_norm.py)、[stage_g_norm.py](file:///home/shengfeng/Privacy-inference/src/stage_g_norm.py)、[stage_g_ffn.py](file:///home/shengfeng/Privacy-inference/src/stage_g_ffn.py)

常见坑：

- dtype：部分配置在 BF16 下可能出现溢出或 NaN，建议先用 FP32 对齐逻辑，再优化 dtype
- κ（kappa）：不同层、不同位置（input_norm / post_attn_norm）通常需要单独估计，否则误差会积累

## 模块化引擎（src/aloepri/）

`src/aloepri/` 是“从分阶段脚本”抽象出来的可复用模块层，目标是让接手者更容易：

- 以配置驱动方式组装 AloePri（而不是复制 stage 脚本的粘合代码）
- 复用 attention/norm/ffn/embed/head 的封装
- 在不破坏既有 Stage 回归脚本的情况下迭代架构

核心文件：

- 配置对象： [config.py](file:///home/shengfeng/Privacy-inference/src/aloepri/config.py)
- 引擎（模型改写/组装入口）： [engine.py](file:///home/shengfeng/Privacy-inference/src/aloepri/engine.py)
- 层封装（attention/norm/ffn/embeddings）：[layers/](file:///home/shengfeng/Privacy-inference/src/aloepri/layers/)

当前定位：

- `AloePriEngine` 通过复用 Stage G/H 的“融合版实现”来构建可运行链路
- 它是“可接手的工程抽象层”，并不替代 `scripts/run_stage_*.py` 的实验进度管理脚本

## 常见问题（FAQ）

### 1) 为什么模块化引擎跑全层时要用 FP32？

部分随机变换（KeyMat、噪声、融合后的权重）在 BF16 下更容易出现溢出或 NaN。建议：

- 先用 FP32 让“索引/变换/融合逻辑”对齐
- 再逐步切换 BF16，并在 `outputs/` 中观察误差是否呈可控增长

### 2) perm_vocab / inv_perm_vocab / restore_logits 到底怎么用？

- 输入映射：用 `perm_vocab[input_ids]`（见 [transforms.py](file:///home/shengfeng/Privacy-inference/src/transforms.py)）
- 输出还原：优先用 `restore_logits(logits_perm, perm_vocab)`；不要手写 `[:, :, inv_perm_vocab]`，以免搞错“索引选择”和“逆置换”的方向

### 3) 想新增模型或新增 attention profile 应该改哪里？

- 新增 attention profile：在 [attention_keys.py](file:///home/shengfeng/Privacy-inference/src/attention_keys.py) 扩展 profile，并确保 [stage_h_attention_static.py](file:///home/shengfeng/Privacy-inference/src/stage_h_attention_static.py) 的静态化路径能消费这些 keys
- 新增模型结构：优先从 `src/aloepri/layers/` 拆出模型特定的 adapter（例如 Qwen2 的 attention/ffn/norm），避免把模型特例散落在 stage 脚本里
