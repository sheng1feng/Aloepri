# Qwen 安全评测执行计划（闭环版）

## 1. 计划目标

本计划用于把 `Qwen` 安全评测工作从“方向明确”推进到“可以逐阶段执行、逐阶段验收”的状态。

这份计划的要求是：

- **闭环**：每个阶段都有输入、动作、输出、验证、退出条件；
- **可验证**：每个阶段都有明确的脚本、文档、JSON、测试或表格作为证据；
- **可串行推进**：前一阶段未达标，不随意进入后一阶段；
- **可回看**：后续任何一次实验结果都能挂到统一矩阵、统一 schema、统一报告框架中。

一句话理解：

> 这不是“要做什么”的愿望单，而是“每一步做完以后怎样判断真的做完”的执行蓝图。

---

## 2. 执行原则

## 2.1 关卡制推进

后续工作统一按 `Gate` 推进：

- `Gate 0`：Phase 0 真正闭环
- `Gate 1`：VMA 最小实现闭环
- `Gate 2`：IMA 最小实现闭环
- `Gate 3`：ISA 最小实现闭环
- `Gate 4`：TFMA / SDA 闭环
- `Gate 5`：参数-安全性联合扫描闭环
- `Gate 6`：增强点验证与首版安全报告闭环

规则：

- 未通过当前 Gate，不进入下一 Gate；
- 允许修补当前 Gate；
- 不允许“脚本没稳定、结果没沉淀、报告没挂接”就推进到下个攻击。

## 2.2 统一证据格式

每个 Gate 的完成必须至少留下以下三类证据：

1. **代码证据**
   - `src/`
   - `scripts/`
   - `tests/`

2. **结果证据**
   - `outputs/security_qwen/...`

3. **文档证据**
   - `docs/qwen_security/...`
   - `推进看板` 更新

## 2.3 统一口径

后续所有攻击实验都必须回答相同四个问题：

1. 攻击输入是什么；
2. 攻击恢复目标是什么；
3. 主指标是什么；
4. 在 `Stage H / J / K` 上的风险排序是什么。

---

## 3. 当前基线快照

截至当前，已经完成的基础工作有：

- 安全工作目录已建立；
- 总计划、Phase 0 文档、推进看板已建立；
- `src/security_qwen/` 空骨架已建立；
- `scripts/security_qwen/` 空 runner 已建立；
- 统一 JSON schema 已建立；
- 首版评测矩阵已建立；
- 基础测试已通过。

这意味着：

> 我们已经完成了“计划框架层”的第一步，但 **Phase 0 还没有完全闭环**，因为还缺少聚合器、统一汇总流程、样例结果导出规范与部分实际 runner 约束。

---

## 4. 总体执行顺序

建议的严格执行顺序如下：

1. **先补完 Gate 0**
   - 把 Phase 0 从“骨架已搭好”推进到“真正可作为后续攻击统一底座”

2. **再做 Gate 1 / VMA**
   - 先拿到最直接的静态权重攻击结果

3. **再做 Gate 2 / IMA**
   - 验证训练式反演是否比静态匹配更危险

4. **再做 Gate 3 / ISA**
   - 聚焦 client/server 分离部署后的中间态攻击

5. **再做 Gate 4 / TFMA + SDA**
   - 测试长期在线统计泄露

6. **再做 Gate 5 / 参数扫描**
   - 形成 accuracy-privacy 曲线

7. **最后做 Gate 6 / 增强点验证**
   - 在已经知道“哪里最危险”的前提下做加强

这个顺序的原因是：

- `VMA / IMA / ISA` 最有可能最早发现主风险；
- `TFMA / SDA` 更偏统计长期攻击，适合后置；
- 增强点必须建立在“风险主次已明确”的前提上。

---

## 5. Gate 0：Phase 0 真正闭环

## 5.1 目标

把当前已有的 Phase 0 文档与骨架，补成后续所有攻击都可直接复用的统一底座。

## 5.2 输入

- `docs/qwen_security/安全评测总计划.md`
- `docs/qwen_security/Phase0_安全基线与实验约定.md`
- `src/security_qwen/`
- `scripts/security_qwen/`

## 5.3 工作包

### Work Package 0.1：统一目标解析

任务：

- 统一 `Stage H / J / K` 目标解析接口；
- 明确 release profile 如何映射到真实 artifact；
- 为后续 runner 提供一致的 target loader。

交付：

- `src/security_qwen/artifacts.py` 增补解析能力
- 至少一个 target resolve 测试

验收：

- 任一 runner 只需要 `target name` 即可拿到统一 target 描述

### Work Package 0.2：统一结果聚合

任务：

- 新增 phase0 汇总器；
- 能扫描 `outputs/security_qwen/*/*.json`；
- 自动汇总成 summary catalog。

交付：

- `src/security_qwen/summary.py`
- `scripts/security_qwen/export_security_summary.py`
- `outputs/security_qwen/summary/security_catalog.json`

验收：

- 模板结果文件可被正确聚合

### Work Package 0.3：统一样例结果

任务：

- 为 `VMA / IMA / ISA / TFMA / SDA` 各导出至少 1 个模板结果；
- 确认不同攻击的结果都能通过 schema 校验；
- 确认都能被汇总器读取。

交付：

- 每类攻击至少 1 个模板 JSON

验收：

- 模板文件全部可解析、可汇总

### Work Package 0.4：统一测试

任务：

- 为 schema、matrix、target resolving、summary aggregation 增加测试；
- 保证 Phase 0 资产本身是稳定的。

交付：

- `tests/test_security_qwen_*.py`

验收：

- 最小测试集全部通过

## 5.4 Gate 0 输出

- 统一 schema
- 统一 matrix
- 统一 target loader
- 统一 summary exporter
- 统一模板样例
- 统一测试集

## 5.5 Gate 0 验证命令

建议最终至少包含：

```bash
conda run --no-capture-output -n qwen-transformers python -m pytest -q \
  tests/test_security_qwen_schema.py \
  tests/test_security_qwen_matrix.py \
  tests/test_security_qwen_summary.py

conda run --no-capture-output -n qwen-transformers python scripts/security_qwen/export_security_matrix.py
conda run --no-capture-output -n qwen-transformers python scripts/security_qwen/export_security_summary.py
```

## 5.6 Gate 0 通过标准

- [ ] 所有攻击模板都通过统一 schema 校验
- [ ] 统一矩阵可导出
- [ ] 统一汇总可导出
- [ ] 目标解析无歧义
- [ ] Phase 0 文档、看板、代码三者一致

## 6. Gate 1：VMA 最小实现闭环

## 6.1 目标

做出第一版可运行的 `VMA` 攻击，先得到最基础的 token permutation 恢复率。

## 6.2 工作包

### Work Package 1.1：权重对定义

- 明确哪些权重对作为首轮 VMA 输入
- 先做 embedding/head 相关组合
- 再扩展到 attention / ffn 相关组合

### Work Package 1.2：匹配基线

- 实现 row-sort / neighbor matching 基线
- 输出 token Top-1 / Top-k 恢复率

### Work Package 1.3：跨层投票

- 引入多层结果投票
- 输出 voting 前后差异

### Work Package 1.4：工件对比

- 至少覆盖：
  - `stage_h_full_obfuscated`
  - `stage_j_stable_reference`
  - `stage_j_tiny_a`
  - `stage_k_stable_reference`
  - `stage_k_tiny_a`

## 6.3 Gate 1 输出

- `src/security_qwen/vma.py` 真实实现
- `scripts/security_qwen/run_vma.py` 可跑
- `outputs/security_qwen/vma/*.json`
- `docs/qwen_security/` 中的 VMA 阶段报告

## 6.4 Gate 1 验收标准

- [ ] 至少一个 target 上能跑通完整 VMA
- [ ] 输出符合统一 schema
- [ ] 有 token Top-1 / Top-k 恢复率
- [ ] 有 `Stage H/J/K` 横向对比
- [ ] 能回答“哪类工件更易被恢复”

---

## 7. Gate 2：IMA 最小实现闭环

## 7.2 工作包

- 明确 inversion 训练文本
- 划分 `train / val / test`

### Work Package 2.2：最小反演器

- 再做 embedding regression 头

### Work Package 2.3：工件对比

- `Stage A`
- `Stage H`
- `Stage J stable_reference`
- `Stage J tiny_a`

## 7.3 Gate 2 输出

- `src/security_qwen/ima.py` 真实实现
- `scripts/security_qwen/run_ima.py`
- `outputs/security_qwen/ima/*.json`

## 7.4 Gate 2 验收标准

- [ ] 至少一个模型反演器能稳定训练
- [ ] 有 Top-1 / Top-k 恢复率
- [ ] 有 embedding cosine 指标
- [ ] 能回答“当前训练式反演是否比 VMA 更危险”

## 8. Gate 3：ISA 最小实现闭环

## 8.1 目标

围绕“真实 client/server 分离部署下可见中间量”，做最小中间态攻击闭环。

## 8.2 工作包

### Work Package 3.1：observable 接口冻结

- 明确首轮只做哪些 observable：
  - `hidden_state`
  - `layer_output`
  - `attention_score`
  - `kv_cache`

### Work Package 3.2：最小恢复器

- 至少一种 observable 上做 token 恢复
- 先做单层，再做多层

### Work Package 3.3：部署线对比

- `Stage H`
- `Stage J`
- `Stage K`

## 8.3 Gate 3 输出

- `src/security_qwen/isa.py` 真实实现
- `scripts/security_qwen/run_isa.py`
- `outputs/security_qwen/isa/*.json`

## 8.4 Gate 3 验收标准

- [ ] 明确了每种 observable 的攻击接口
- [ ] 至少一种 observable 攻击可跑通
- [ ] 有中间态 token 恢复率
- [ ] 能回答“部署态最危险的中间量是什么”

---

## 9. Gate 4：TFMA / SDA 闭环

## 9.1 目标

回答 deterministic token permutation 在长期在线统计下会泄露多少信息。

## 9.2 工作包

- `TFMA` 首版实现
- `SDA` 首版实现
- 三种知识设置：
  - `zero_knowledge`
  - `domain_aware`
  - `distribution_aware`

## 9.3 Gate 4 验收标准

- [ ] `TFMA` 有 Top-10 / Top-100 恢复率
- [ ] `SDA` 有 BLEU-4
- [ ] 三种知识设置齐全
- [ ] 能回答“频率攻击在 release 工件上是否构成主要风险”

---

## 10. Gate 5：参数-安全性联合扫描

## 10.1 目标

把 correctness-only 的工作点选择，升级成 accuracy-privacy tradeoff。

## 10.2 工作包

- 固定一小组代表性参数：
  - `alpha_e`
  - `alpha_h`
  - 必要时 `lambda`
- 在相同 target 上复用：
  - `VMA`
  - `IMA`
  - `ISA`

## 10.3 Gate 5 验收标准

- [ ] 至少形成一张参数-安全性表
- [ ] 至少形成一张 accuracy-privacy 曲线
- [ ] 给出当前推荐工作点

---

## 11. Gate 6：增强点验证与首版安全报告

## 11.1 目标

在风险主次已经明确后，验证最值得做的 1~2 个安全增强点。

## 11.2 工作包

- 选择 1~2 个增强点，例如：
  - 更细粒度噪声点
  - 频率侧轻量平滑策略
  - profile 交付分层
- 跑前后对比
- 输出首版安全报告

## 11.3 Gate 6 验收标准

- [ ] 至少一个增强点有可量化收益
- [ ] 首版安全报告完成
- [ ] 看板更新为“首轮安全评测闭环完成”

---

## 12. 文档与代码同步要求

后续每完成一个 Gate，必须同步更新：

1. `docs/qwen_security/推进看板.md`
2. 对应阶段文档
3. 对应输出目录
4. 对应测试

任何只改代码、不改文档、不留输出的阶段，都不视为闭环完成。

---

## 13. 当前建议的立即动作

当前最合理的下一步不是直接写攻击算法，而是：

1. 先补完 `Gate 0`
2. 再按顺序进入：
   - `Gate 1 / VMA`
   - `Gate 2 / IMA`
   - `Gate 3 / ISA`

原因：

- 这三类攻击最能最快建立主风险排序；
- 现在基础骨架已搭好，最需要的是把 Phase 0 的聚合与验收链补齐；
- 否则后续即使攻击脚本能跑，也很难形成统一结论。

---

## 14. 一句话结论

> 接下来的执行策略必须是“先闭环 Gate 0，再逐个击穿 VMA、IMA、ISA 三个主风险入口”，并要求每个 Gate 都留下脚本、测试、JSON 结果和文档四类证据，只有这样整个 `Qwen` 安全评测工作才是真正可执行、可验证、可收敛的。
